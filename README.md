[] В целом нет необходимости усложнять архмтектуру лишними классами или компонентами, если можно сделать таким образом, чтобы вместо жёстко связанной логики было легко адаптируемые обьекты. Для этого предлагается контролировать UI через отрисовку поверх, но двигать при этом обьекты с камерой (вернее подвязать на событие движения камеры). Почему такой подход - потому что это позволит сделать динамичность элементов интерфейса, без необходимости переподключать иерархию и т.д. Все выше компоненты будут быть как обьекты мира, хранящихся в пресохраненных обьектах (ObjectTemplate/ObjectModel) и быть как Behavior- обьекты поведения. Component-ы при этом содержат данные например.

[×] Transform - структура, хранящая Position, Rotation, Scale структуры, производные от Vector2 / Vector3, где значения сохраняются в локальном пространстве. Используется для определения объектов в пространстве.

[×] Collider - структура, хранящая Polygon2 / Polygon3, где значения сохраняются в локальном пространстве. Используется для 

взаимодействия обьектов друг с другом, а также обьектов с точками.

[] Mask - структура, хранящая Collider стуруктуры (входное значение и кеш), где значения сохраняются в локальном пространстве. Используется для корректировки визуализации объектов.

[×] Renderer - структура, хранящая стек спрайтов. Используется для визуализации объектов в пространстве.

[×] ObjectTemplate<> - обьект, являющийся образцом для наполнения.

[×] ObjectModel<> - обьект, являющийся образцом для клонирования.

[×] ObjectTemplate Button - образец кнопки, требует наличие трансформы, колайдера, отрисовщика (спрайт кнопки в активном / неактивном состоянии и спрайт кнопки в нажатом состоянии). Включает защищённые делегаты на нажатие миши (OnMouseDown, OnMouseHold, OnMouseUp), публичные делегаты на нажатие (OnClick).

[×] ObjectTemplate SliderBlock - образец слайдера, требует наличие трансформы, отрисовщика (спрайт слайдер поля в активном / не активном состоянии), слайдер линию (две трансформы / точки). Включает SliderBar, публичный делегаты на обновление слайдера (OnValueWrote, OnValueChanged).

[×] ObjectTemplate SliderBar - образец ползунка слайдера, требует наличие трансформы, колайдера, отрисовщика (спрайт ползунка слайдера в активном / не активном состоянии и нажатом состоянии). Включает защищенные делегаты на нажатие и движения мыши (OnMouseDown, OnMouseHold, OnMouseMove, OnMouseUp), публичные делегаты на нажатие и движение мыши (OnClick, OnMove).

[×] ObjectTemplate CheckButton - образец флага, требует наличие трансформы, колайдера, отрисовщика (спрайт флага в активном /не активном состоянии, спрайт галочки в активном / не активном состоянии). Включает публичные делегаты на обновление флага (OnEnabled, OnDisabled, OnSwitched).

[×] ObjectTemplate RadioButton - образец переключателя, требует наличие трансформы, колайдера, отрисовщика (спрайт кнопки в активном / не активном состоянии, спрайт галочки в активном / не активном состоянии), RadioButtonGroup (контейнер переключателей). Включает защищённые делегаты на нажатие мыши (OnMouseDown, OnMouseHold, OnMouseUp), публичный делегат на нажатие мыши (OnClick), публичные делегаты на обновление переключателя (OnEnabled, OnDisabled, OnSwitched).

[×] ObjectTemplate Block - образец блока / формы (контейнера для других элементов), требует наличие трансформы, колайдера, отрисовщика (спрайт блока / формы в активном (или сфокусированном) / не активном (или расфокусированном) состоянии), маски (опционально), слайдера (опционально). [] Концепция связанная с аналогией в космосе. Главный обьект (инстанс приложение) - это Space. Основной составной единицей является World (мир), который состоит из Location (уровней). Локация является минимальной составной единицей, в которой сохраняются обьекты (суб составные единицы). Чтобы было проще понять, вот пример: При запуске движка, создается пространство, пространство обязано быть заполнено чем-то. И этим чем-то является по умолчанию движковый мир, в котором есть куча локаций (разные менюшки, панельки, окна). Учитывая, что в пространстве, каждый действующий мир обрабатывается, а в каждом мире обрабатывается каждая действующая локация, и при этом каждая локация может иметь свой вьюпорт - мы имеем, чёткую иерархию, позволяющую компоновать миры и локации в пронстранстве в пределах одного, или нескольких окон (окно не стоит принимать, чем-то особенным). Таким образом, в пространстве позволено динамически создавать/менять новые/старые миры и локации. Даже движковый мир может меняться. Запуская движок действующий миров будем как минимум один (движковый мир создается автоматически и устанавливается как стартовый). А запуская игру действующий мир будет тот, который сконфигурирован ранее как стартовый. Таким образом не только в движке может быть несколько действующих миров (движковый и игровые), а и в самой игре. Окно само по себе не является вьюпортом, а лишь холстом, на котором размещаются вьюпорты. Если одного окна не достаточно, можно создавать дополнительные окна и размещать в них остальные вьюпорты. Вьюпорт может отображаться внутри другого вьюпорта, путем отрисовки поверх него или накладываться, если альфа канал позволяет.
