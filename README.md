Структура папок движка:
- Eightgine - Содержит 3 ключевые папки.
    - Binaries - Содержит исполняемые файлы и шаред библиотеки. Исполняемый файл, сгенерированный сборкой в зависимости от опции WITH_EDITOR, будет запускать пустой проект с редактором и без него.
    - Development - Содержит файлы необходимые для разработки проекта. Содержит шаблон CMakeLists.txt файла для сборки проекта.
        - Engine - Содержит исходный код движка и редактора.
        - Game - Содержит шаблон исходного кода для сборки исполняемого файла проекта.
        - ThirdParty - Содержит исходные файлы сторонних библиотек, фреймворков, необходимых для движка и редактора.
        - Intermediate - Содержит артефакты сборки и промежуточные файлы. Может быть удалена.
    - Resources - Содержит сереализированные данные, ассеты редактора.

Структура папок проекта:
- Project - Содержит 3 ключевые папки. Может иметь произвольное имя.
    - Binaries - Содержит исполняемые файлы и шаред библиотеки. Исполняемый файл, сгенерированный сборкой в зависимости от опции WITH_EDITOR, будет запускать проект с редактором и без него.
    - Development - Содержит файлы необходимые для разработки проекта. Содержит копию шаблона CMakeLists.txt файла для сборки проекта. Может изменятся разработчиком.
        - Game - Содержит исходный код проекта. Содержи копию шаблона исходного кода для сборки исполняемого файлы проекта.
        - ThirdParty - Содержит исходные файлы сторонних библиотек, фреймворков, необходимых для проекта.
    - Resources - Содержит сереализированные данные, ассеты проекта.

- Project - Содержит 2 ключевые папки. Может иметь произвольное имя.
    - Binaries - Содержит исполняемые файлы и шаред библиотеки.

Прежде чем спроектировать главный цикл движка стоит обсудить компоненты, которые будет включать любая 2D игра:
1. Спрайты - Весь визуал игры строится на спрайтах. С их помощью, мы можем реализовать графический интерфейс, игровое окружение, анимации и т.п;
2. Звук - Является вторым, после визуального восприятия, элементом игрового опыта. С его помощью, мы можем реализовать динамику, отзывчивость, сопровождение и т.д.
3. Ввод - Благодаря возможности ввода, мы можем контролировать игровой процес, управлять персонажами, взаимодействовать с игрой и т.п;
4. Игровые ресурсы - В общем смысле, включат картинки, звук и другие файлы, поддающиеся кастомизации. Благодаря доступу к ресурсам игры, открывается возможность для создания модов и дополнений;
5. Игровая логика - Самое важное, что связывает остальные 4 компоненты, благодаря ей строится весь игровой процес.

Имея полное представление об компонентах, можно распределить их обработку так, чтобы они не мешали друг другу и могли свободно масштабироваться. Первым делом, после запуска игры может произойти одна из 3-х вещей:
1. Экран загрузки игровых ресурсов (и загрузка игровых ресурсов) -> отображение игрового меню;
2. Экран со вступительными картинками и видео (и загрузка игровых ресурсов) -> отображение игрового меню;
3. Отображение игрового меню (и загрузка игровых ресурсов).

Вышеперечисленные вещи могут комбинироваться, но унифицировать поведение в рамках движка нельзя, поэтому предлагается гибкий подход. Базовая архитектура обьектов движка состоит с пространства и миров, где миры в свою очередь включат в себя локации. Представим, что первым миром в игре будет мир с загрузочным экраном, или же локация. Загрузка игровых ресурсов может выполнятся динамически, во время игры, например, при загрузке пространства, мира, локации. И пространство, и мир, и локация имеют ресурсы, необходимые для их выполнения. Ресурсы, распространяющиеся на несколько миров, хранятся единичным экземпляром в пространстве. Ресурсы, распространяющиеся на несколько локаций, хранятся в единичном экземпляре в мире. Любой активированный мир или локация автоматически подтягивает загрузку ресурсов, при этом не исключается распространение подгрузки ресурсов через связи. Например, если А связан с Б, а Б связан с В, то при загрузке А, можно дополнительно подгрузить Б или даже В.
Иходя из этого, в первую очередь, при запуске игры создается пространство, после которого с конфигурационного файла условно загружается мир, в чьих правах контролировать процес загрузки ресурсов и локаций. Локация сама по себе является полноценной игровой площадкой, и следовательно для игры с открытым миром будет достаточно 2-3 локации (одна для главного меню, вторая для игрового уровня, третья в качестве вспомогательной).
Таким образом, в главном цикле движка на первой позиции стоит инициализация корневых подсистем (память, файлы, потоки), после которой уже идёт загрузка конфигурационного файла, с которого начинается инициализация базовых подсистем (отрисовка, звук, ввод, сеть), а затем загрука мира. Далее по ходу выполнения подгружаются остальные подсистемы (физика, анимации, освещение, скриптование). Но поскольку корневые подсистемы также требуют конфигурационных данных, их передача выполняется средставами системы сборки. Учитывая, что при сборке игры требуется также собирать движок, эти конфигурационные данные могут регламентироваться самим разработчиком, если значения по умолчанию его не удовлетворяют.

Когда загрузка и активация мира выполнена со всеми нужными подсистемами, главный цикл движка начинает обработку активированного мира и его локакий. Активацию и деактивацию разрешено выполнять в главном цикле, которая применяется на следующем тике. Заметим, что подсистемы ввода, отрисовки, звука, файлов, сети, игрового логики работают паралельно. Тоесть, когда выполняется загрузка уровня, мы можем спокойно можем посылать ввод, отрисовывать окно загруки, проигрывать звук, или вообще подгружать соседние локации. Поскольку корневая подсистема потоков хранит резервные потоки, их можно использовать для собственных целей, если это необходимо.

Порядок обновления базовых подсистем должен выполняется в строгом порядке, но чтобы понять, что за чем следует, разобраться с предназначением подсистем:
1. Ввода - В любой момент времени, игрок может посылать события с устройств ввода, например: мышка, клавиатура, джойстик. Но обрабатывать события ввода мы можем только на фиксированном временном шаге (тике). Следовательно, чтобы игра была отзывчивой, стоит обрабатывать ввод в первую очередь.
2. Игровая логика - Отвечает за поведение игровых обьектов, в зависимости от ввода игрока и других событий. Всегда работает на серверной стороне. В случае одиночной игры сервером выступает устройство, на которой запущена игра.
3. Звук - Воспроизводит звуковую составляющую. Если мы хотим всегда получать актуальное воспроизведение, необходимо обновлять после его обработки игровой логики. 
4. Графика - Отображает визуальную составляющую. Если мы хотим всегда получать актуальное отображение, необходимо обновлять его после воспроизведение звука (из-за буферизации аудио, которая имеет задержки).

Отметим, что игровая логика выполняется только на сервере, который принимает данные от клиентов, обрабатывает их и отправляет. Каждый клиент получает данные на основании которых проигрывается звук, отрисовывается графика. Архитектурно клиент имеет буфер для отправки данные на сервер, а также буфер для их получения. Сервер при это хранит буфера для получения данные от клиентов, а также для отправки. Эти буфера хранятся в подсистеме Сети.
